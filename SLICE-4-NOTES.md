## Surface added

```text
GET /api/readings/range?start=ISO&end=ISO&bucket=raw|hour|day|week
  → 200 BucketedReadingsResponse
  → 400 ProblemDetails { error: "invalid_start" | "invalid_end" | "invalid_bucket" | "start_after_end" | "range_too_large", message, requestId }

GET /api/readings/heatmap?year=YYYY
  → 200 HeatmapResponse (always 365 or 366 cells)
  → 400 ProblemDetails { error: "missing_year" | "invalid_year", message, requestId }
```

Both endpoints cursor-clip with `WHERE ReadingTime <= @asOf` where
`@asOf` comes from the scoped `CursorSnapshot` (slice 1 plumbing).

## What was deliberately NOT built (deferred to later slices)

* **Anomaly overlays on the time-series** — slice 9 (#11) lays the
  anomaly detection pipeline; rendering markers on the Explorer chart
  follows.
* **Forecast bands / predicted-T trace** — slice 7 (#9) ships the
  lag-LR forecaster; the Explorer can subscribe to `Forecasts` once
  that lands.
* **Cursor-aware auto-refresh** — slice 12 (#14) wires the
  `clock-changed` SSE event to a browser handler that re-fetches.
  Slice 4 reads once per UI interaction.
* **Range-selector inside the heatmap** — clicking a heatmap cell
  could (and should) re-target the time-series chart at that day. The
  state-machine ergonomics for that interaction were judged out of
  scope for slice 4; tracked as a future polish ticket.
* **Server-side bucket boundary literal exposure** — the `bucketTime`
  in aggregated responses is always UTC and aligned on the bucket's
  origin. We do not yet expose the bucket *width* in seconds for
  consumers who want to chart per-bucket means; the consumer can
  derive it from the `bucket` enum.
* **Comfort / alert / leaderboard panels** — slices 8 / 11 / 12.

## Judgment calls

1. **`raw` cap = 7 days (default).** The spec asked for a documented
   policy; 7 days at 1-minute resolution = ~10 080 rows, ~1 MB JSON,
   well within budget for a single fetch. The cap is overridable via
   `CLIMASENSE_RAW_MAX_DAYS`. An alternative — capping by row count
   rather than time-span — was rejected because the row count requires
   a probe query before the SQL select fires. Time-span is a constant
   guard, evaluated client-side of SQL.

2. **CDN Plotly vs vendored.** Plotly minified is ~3 MB. CDN is the
   pragmatic choice for the portfolio demo and is documented in
   `plotly-config.js`. The single `<script src="...cdn.plot.ly...">`
   line is the only swap to vendor offline.

3. **Bucket alignment matches SQL Server's default origin
   (1900-01-01, a Monday).** The .NET densification's `AlignDown`
   mirrors that exactly. An alternative — passing an explicit origin
   to `DATE_BUCKET` — was rejected because it would need a matching
   client-side change and no business reason to deviate.

4. **`RangeFetcher` + `HeatmapFetcher` as two delegates, not one
   `IReadingsRepository` interface.** Follows ADR-0011 — two seams,
   two delegates, no speculative interface. When a third reading-side
   read appears (e.g. live forecast reads) the interface might cohere
   from the three shapes.

5. **Empty aggregated buckets are filled in .NET, not SQL.** SQL
   Server's `GENERATE_SERIES` + LEFT JOIN could push the work down
   but adds query complexity for no measurable benefit at this slice's
   response sizes (worst case ~8 760 `hour` buckets over a year).
   The pure-CLR densification is straightforward to unit-test and the
   `DensifyBuckets` helper is internal-visible.

6. **End past cursor is silently clamped to the cursor (not a 400).**
   Two reasons:
   (a) Under WallClock, the cursor IS `DateTime.UtcNow`, so an end of
       "now + 5 min" is a benign UI rounding artifact (Date.now in
       JS) — rejecting it would inflict a 400 on the user for no
       useful diagnostic.
   (b) Under ReplayClock (slice 12), a reviewer scrubbing the cursor
       backwards would otherwise have to manually clamp every range
       query. Silent clipping matches the slice-3 `/api/readings/latest`
       behaviour (which also clips at the cursor).

7. **PageModel is empty.** No server-side state for the Explorer; the
   page is a single Razor template + 100 % client-side data hydration.
   This matches the slice-3 dashboard pattern and lets us share the
   same WebApplicationFactory test harness.

8. **The new endpoints add to the FastAPI ContractValidator's
   exclusion sets.** Same rationale as slice 3's
   `/api/readings/latest`: the .NET tier serves them, the FastAPI ml
   tier never does, so excluding them from the parity check is
   correct — not a workaround.
